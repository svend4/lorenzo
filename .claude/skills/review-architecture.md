# review-architecture

Архитектурное ревью документа / раздела / системы.

## Когда использовать

- "проверь архитектуру Svyazi"
- "сделай review дизайна X"
- "найди проблемы в спецификации Y"
- "что не так с этим компонентом"

## Шаги

1. **Определить scope ревью**
   - Один документ? Ансамбль? Раздел `docs/svyazi-2-0/architecture/`?

2. **Прочитать всё в scope**
   - Тексты документов
   - Связанные ADR (`DECISIONS.md`)
   - Связанные риски (`RISK_REGISTER.md`)

3. **Чеклист ревью** (по 10 осям)

   | Ось | Что проверять |
   |---|---|
   | **Цели и метрики** | Есть ли SMART-цели? Метрики успеха? |
   | **Контракты** | Чёткие input/output контракты между компонентами? |
   | **Failure modes** | Что делает система при отказе компонента? |
   | **Privacy / Security** | PII, секреты, attack surface |
   | **Observability** | Логи, метрики, трейсинг |
   | **Versioning** | Как мигрировать формат данных, API? |
   | **Backward compat** | Что ломается при upgrade? |
   | **Cost model** | Сколько стоит scale в 10x? |
   | **Vendor lock-in** | Можно ли заменить провайдера / БД? |
   | **Test strategy** | Как тестировать end-to-end? |

4. **Идентифицировать антипаттерны**

   Для Svyazi 2.0 особенно:
   - **God component:** один компонент знает всё
   - **Sync chain:** длинная синхронная цепочка вызовов (отказы каскадируют)
   - **Hidden coupling:** компоненты передают через global state
   - **Schema drift:** формат данных не задокументирован
   - **Missing middle layer:** см. Layer-B paper в nautilus

5. **Сформировать отчёт**

   ```markdown
   # Архитектурное ревью: [Название]

   ## Сводка
   **Балл:** [X/10] по 10 осям
   **Критичные проблемы:** N
   **Рекомендации:** M

   ## По осям
   | Ось | Балл | Заметки |
   |-----|------|---------|
   | Цели | 8/10 | … |
   | …

   ## Антипаттерны
   - [Антипаттерн 1] в [`файл.md`]: …

   ## Рекомендации (приоритизованные)
   ### Критичные (блокирующие)
   1. [Рекомендация 1]
       - Почему: …
       - Как: …

   ### Важные (стоит сделать)
   2. …

   ### Косметические
   3. …

   ## Open questions для автора
   - ?
   ```

6. **Если найдены противоречия** — вызвать `find-contradictions`

7. **Если выявлены риски** — создать запись:
   ```bash
   python scripts/improve_template_init.py --type risk-entry \
     --slug docs/risks/RISK-NNNN.md
   ```
