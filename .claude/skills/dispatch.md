# dispatch

Запуск нескольких скилов параллельно или последовательно с агрегацией результата.

## Когда использовать

- Когда задача требует нескольких независимых действий
- «дай мне отчёт по A и B»
- «проверь X и Y одновременно»

## Шаги

### Параллельная диспетчеризация

Если задачи независимы — выполнить все одновременно (для агента это значит
сделать несколько tool-вызовов в одном сообщении).

Пример: «Дай мне статус проекта Yodoca + список похожих авторов»
- Параллельно:
  - `search_docs(query="Yodoca")` → найти упоминания
  - `get_project_status(name="Yodoca")` → теги, метрики
  - `propose_outreach(author="VitalyOborin")` → черновик письма автору
- Агрегировать в единый отчёт

### Последовательная диспетчеризация

Если задача — пайплайн: A → B → C. Каждый шаг использует результат предыдущего.

Пример: «Найди RAG-документы, выбери топ-3, сделай по ним синтез»
1. `search` query="RAG" → получить топ-10 файлов
2. `summarize` для каждого из топ-3
3. `synthesize` на собранных резюме

## Шаблон работы

1. **Декомпозировать задачу**
   - Определить под-задачи
   - Определить зависимости (какая зависит от какой)

2. **Построить план**
   ```
   Параллельные группы:
   - Группа 1: A, B, C (независимы)
   - Группа 2: D (зависит от A)
   - Группа 3: E (зависит от B, D)
   ```

3. **Выполнить группа за группой**
   - Внутри группы — параллельно
   - Между группами — последовательно

4. **Агрегировать**
   - Объединить выходы в один отчёт
   - Помечать атрибуцию: «из A: …, из B: …»

5. **Если что-то упало — продолжить с остальным**
   - Не блокировать всю задачу из-за одного fail
   - В отчёте указать «X не получилось, причина: …»

## Антипаттерны
- ❌ Запустить всё параллельно когда есть зависимости — некоторые задачи получат пустой контекст
- ❌ Запустить всё последовательно когда нет зависимостей — потеря времени
- ❌ Не агрегировать — пользователь получает 5 отдельных ответов вместо одного
