## 4. Симбиотическая Архитектура

Комбинация InGit и Cowork создаёт архитектуру,
которую ни один не обеспечивает в одиночку.

### 4.1. Декомпозиция по Слоям

Работая снизу вверх:

```
СЛОЙ 0 — Аппаратное обеспечение и ОС
  Локальный компьютер, опционально NAS
                ↓
СЛОЙ 1 — Файловая Система
  Хранение файлов операционной системы
                ↓
СЛОЙ 2 — Подложка InGit
  Структурированные папки (00_inbox через
  90_exports)
  YAML схемы метаданных
  Git-нативные конвенции версионирования
  Хуки валидации
  Шифрование для чувствительных папок
                ↓
СЛОЙ 3 — Cowork Агент
  Постоянная память, ограниченная Project
  Многошаговое выполнение задач
  MCP интеграция коннекторов
  Способность использования компьютера
  Запланированные повторяющиеся задачи
                ↓
СЛОЙ 4 — Пользователь и Внешние Инструменты
  Прямое взаимодействие пользователя с файлами
  Внешние сервисы (GitHub, Notion и т.д.)
  Другие AI-инструменты при необходимости
  Другие члены команды (ограничено; Cowork
  локален)
```

Каждый слой имеет ясные обязанности. Каждый
слой обеспечивает абстракции для слоёв выше.

### 4.2. Как они взаимодействуют

Типичный рабочий процесс комбинирует их:

**1. Пользователь открывает Cowork Project,
указывающий на InGit папку.**

**2. Пользователь заявляет намерение**:
«Напишите черновик раздела о X на основе
существующих материалов.»

**3. Cowork агент начинает**:
- Читает структуру InGit Project (знает
  конвенцию)
- Ищет в `80_wiki/` соответствующие
  существующие знания
- Читает упомянутые документы в
  `30_documents/`
- Составляет новое содержание, следуя шаблону
  документа InGit
- Добавляет соответствующие YAML метаданные
- Валидирует через MCP сервер InGit (если
  настроен) или встроенно
- Сохраняет в соответствующее место
- Обновляет временную линию в `20_timeline/`,
  если конвенция требует
- Коммитит с конвенциональным сообщением
  коммита

**4. Пользователь рецензирует**:
- Видит черновик в правильном месте
- Рецензирует метаданные
- Одобряет или запрашивает ревизии

**5. Итерация продолжается** с Cowork,
помнящим контекст между сессиями.

Это фундаментально отличается от любого из:
- Только чат: нет постоянства за пределами
  чата, нет структуры
- Только Cowork с общей папкой: нет конвенций
  структуры
- Только InGit: нет агентского интеллекта

### 4.3. Где каждый работает

Подложка InGit работает всякий раз, когда:
- Файлы читаются или пишутся
- Происходят Git операции
- Валидация запускается
- Применяется шифрование
- Применяются конвенции структуры

Cowork Агент работает всякий раз, когда:
- Выполняются многошаговые задачи
- Происходит AI синтез
- Доступ к внешним сервисам
- Память вспоминается
- Запускаются запланированные задачи

Пользователь работает всякий раз, когда:
- Намерение выражено
- Выходы рецензируются
- Решения приняты
- Прямые правки выполнены
- Стратегия установлена

Все три работают одновременно. Никто не
требует постоянного присутствия других
(оффлайн работа продолжается без Cowork; Cowork
может работать с не-InGit папками; пользователь
делегирует обоим).

### 4.4. Общее Состояние

Критическая точка реализации: общее состояние
между InGit и Cowork должно быть
**детерминированным и восстанавливаемым**.

**Детерминизм**: Учитывая то же состояние
проекта и те же инструкции Cowork, результаты
должны быть предсказуемыми. Валидация
предотвращает создание Cowork некорректных
артефактов.

**Восстанавливаемость**: Всё состояние живёт
в файлах (отслеживаемых Git). Даже если
сессия Cowork потеряна, состояние Project
восстанавливаемо из файловой системы. Память
может быть восстановлена воспроизведением
истории Git через Cowork.

Вот почему Git-нативность важна: Git
обеспечивает авторитетное состояние. Память
Cowork — удобство, не авторитет.
