# 5. Четыре пути интеграции в порядке доступности

## 5. Четыре пути интеграции в порядке доступности

Мы определяем четыре пути от наиболее-
немедленных до наиболее-зрелых.

### 5.1. Путь 1 — InGit Project в Cowork (Немедленно)

**Разработка не требуется. Доступно сейчас.**

Шаги:
1. Открыть вкладку Cowork в Claude Desktop
2. Создать новый Project, «Начать с нуля»
3. Указать Project на существующую InGit
   папку (или пустую папку с InGit структурой)
4. Добавить пользовательские инструкции,
   описывающие конвенции InGit
5. Добавить референсные документы,
   объясняющие философию InGit
6. Начать работу

**Что работает**:
- Cowork читает структуру InGit
- Пользовательские инструкции информируют
  Cowork о конвенциях
- Cowork может создавать/редактировать файлы,
  следуя конвенциям
- Постоянная память строит знания Project
- Git операции работают через команды
  оболочки

**Чего не хватает**:
- Нет автоматической валидации (Cowork может
  создать невалидный YAML)
- Нет InGit-специфичных операций
- Ручной Git workflow

**Усилие**: Один день для настройки. Ноль
постоянной разработки.

**Рекомендация**: **Сделайте это сейчас**.
Используйте для текущей серии документов
Nautilus/OKWF. Генерирует немедленное обучение
о том, что работает и чего не хватает.

### 5.2. Путь 2 — InGit MCP Сервер (Краткосрочно)

**Требуется разработка. 2-4 недели для базовой
версии.**

Построить `ingit-mcp-server`, выставляющий
операции InGit:

```
Операции Базы Знаний:
- ingit_search_wiki(query)
- ingit_read_document(path)
- ingit_create_document(template, metadata, content)
- ingit_list_recent_documents(days)
- ingit_get_document_metadata(path)

Операции Задач:
- ingit_list_tasks(filters)
- ingit_create_task(metadata, description)
- ingit_update_task(id, updates)
- ingit_decompose_task(id, subtasks)

Операции Валидации:
- ingit_validate_yaml(content, schema)
- ingit_validate_project_structure(path)

Git Операции:
- ingit_commit(message, files)
- ingit_get_recent_changes(days)
- ingit_branch(name, from)

Операции Project:
- ingit_get_project_structure()
- ingit_create_subproject(name, type)
```

Пользователи Cowork добавляют этот MCP сервер
в свой Project. Cowork вызывает эти инструменты
вместо общих файловых операций.

**Что работает, чего нет в Пути 1**:
- Валидация применяется автоматически
- InGit-специфичные операции доступны
- Доменные знания инкапсулированы в сервере
- Более чистые инструкции Cowork (меньше
  объяснения конвенций)

**Усилие**: 2-4 недели разработки. Open source
релиз.

**Рекомендация**: **Начните после месяца
использования Пути 1**. Реальное использование
информирует дизайн MCP инструментов. Избегайте
преждевременного дизайна.

### 5.3. Путь 3 — InGit Cowork Project Шаблон (Среднесрочно)

**Требует, чтобы Anthropic выставил API шаблонов
Project публично.**

Как только Anthropic включит сторонние шаблоны
Project, InGit публикует шаблон:
- Предварительно настроенная структура папок
- Предварительно написанные пользовательские
  инструкции
- Опционально предустановленный MCP сервер
- Образец содержания, объясняющий конвенции
- Документация быстрого старта

Пользователи получают рабочую установку
InGit + Cowork в один клик.

**Что работает, чего нет в Пути 2**:
- Массовое сокращение трения настройки
- Обнаруживаемо пользователями Cowork
- Стандартная конфигурация через пользователей
- Легче делиться между членами команды (когда
  доступно командное разделение)

**Усилие, когда API доступно**: 1-2 недели
для создания шаблона.

**Рекомендация**: **Следить за доступностью
API**. Anthropic, вероятно, включит это; время
неопределено. Быть готовым, когда откроется.

### 5.4. Путь 4 — Глубокая Интеграция Cowork (Долгосрочно)

**Спекулятивно, зависит от созревания API
Cowork.**

Когда Cowork выставит более глубокие API:
- Пользовательские UI элементы внутри Cowork
- Cowork-осведомлённый поток валидации
- Нативная интеграция задач InGit
- Общая память через операции Cowork и InGit
- Кооперативное планирование

Это наиболее амбициозно. Требует, чтобы
Anthropic включил существенное сторонне
расширение. Время полностью вне контроля
автора InGit.

**Усилие, когда возможно**: 3-6 месяцев для
полной интеграции.

**Рекомендация**: **Не планируйте вокруг
этого**. Берите возможности по мере появления.
Не зависьте от этого.

### 5.5. Логика Последовательности

Четыре пути не параллельные варианты. Они
последовательны, каждый строится на предыдущем:

- Путь 1 генерирует **обучение использования**
- Путь 2 строит **автоматизацию** на основе
  обучения
- Путь 3 включает **распространение**, когда
  API доступно
- Путь 4 углубляет **интеграцию**, когда API
  созревает

Прыжок вперёд рискован. Путь 2 без Пути 1
рискует проектированием неправильных
инструментов. Путь 3 без Пути 2 не имеет
ценности для упаковки. Путь 4 без экосистемы
рискует зависимостью от способностей, ещё не
доступных.

Последовательная прогрессия максимизирует
ценность на каждом этапе и откладывает сложные
решения, пока доказательства не доступны.
