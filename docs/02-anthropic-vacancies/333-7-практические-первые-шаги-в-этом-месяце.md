## 7. Практические первые шаги в этом месяце

Конкретные действия для немедленного периода.

### 7.1. На этой неделе

**Действие 1**: Создать тестовый Cowork Project,
указывающий на простую папку с InGit структурой.
Использовать его для одного из восьми
существующих документов серии (например,
переместить `professional-colleague-agents.md` в
доступную для Cowork InGit-стилевую папку).

**Действие 2**: Документировать что работает и
что нет. Конкретно:
- Следовал ли Cowork конвенциям InGit?
- Где он отклонился или испытывал трудности?
- Какие пользовательские инструкции помогли?
- Что было запутанным?

**Действие 3**: Уточнить пользовательские
инструкции на основе наблюдений. Итерировать.

### 7.2. В этом месяце

**Действие 1**: Использовать настройку
Cowork+InGit для **всех восьми документов**
текущей серии. Мигрировать их в надлежащий
InGit проект. Использовать Cowork для
генерации перекрёстных ссылок, выявления
несоответствий, предложения уточнений.

**Действие 2**: Написать конкретные требования
к инструментам MCP на основе месяца
использования. Какие операции InGit Cowork
наиболее выиграл бы от доступа?

**Действие 3**: Набросать начальный дизайн
MCP сервера. Не реализовывать пока; сначала
проектировать.

### 7.3. В следующем квартале

**Действие 1**: Реализовать минимальный MCP
сервер (5-7 инструментов максимум). Тестировать
со своим использованием.

**Действие 2**: Документировать всё для
потенциальных сотрудников.

**Действие 3**: Open source релиз с ясным
объёмом и ожиданиями.

### 7.4. Конкретный Тестовый Случай: Эта Серия Документов

Восемь существующих документов (1, 2, 2.3, 3,
4, 5, 6, 7) обеспечивают идеальный первый
тестовый случай. Они:

- Существенные (~80 000 слов в комбинации)
- С перекрёстными ссылками (Документ 7
  ссылается на 6, 5 и т.д.)
- Естественно версионированные (каждый имеет
  номер версии)
- Разнообразные (технические, концептуальные,
  прикладные)

Миграция их в Cowork-управляемый InGit Project
упражнял бы:
- Конвенции структуры папок
- Управление метаданными
- Обработку перекрёстных ссылок
- Wiki интеграцию
- Git версионирование

Если это работает гладко, подложка
валидирована. Если возникают проблемы, они
раскрывают, что InGit нужно адресовать.

Это «dogfooding» — использование собственного
продукта на собственной работе. Большинство
успешных инфраструктурных проектов начинаются
так.
