## 3. Что InGit обеспечивает, чего Cowork не хватает

Не менее важно: где InGit добавляет ценность,
которую Cowork в одиночку не обеспечивает?

### 3.1. Структурированная Файловая Организация

Cowork Projects могут указывать на любую папку.
Структура папки за пользователем. Без
конвенции структуры варьируются произвольно.

InGit обеспечивает **каноническую структуру**
(00_inbox через 90_exports). Это означает:
- Cowork агент может полагаться на
  согласованный макет
- Работа со знаниями течёт в предсказуемые
  места
- Новые пользователи встречают знакомые
  паттерны
- Шаблоны могут быть разделены
- Кросс-проектное сравнение значимо

Это существенно. Эффективность Cowork
масштабируется с качеством структуры рабочего
пространства. InGit обеспечивает структуру.

### 3.2. Метаданные как Код

Cowork может читать и писать любой формат
файла. Но неструктурированные файлы ограничивают
то, что Cowork может делать. Без метаданных
задачи не имеют статуса, документы не имеют
происхождения, артефакты не имеют отношений.

YAML метаданные как код в InGit обеспечивают:
- Определения задач, которые Cowork может
  парсить и обновлять
- Заголовки документов с версией, автором,
  статусом
- Отношения между артефактами (зависит-от,
  ссылается, происходит-от)
- Схемы валидации, которые Cowork может
  применять
- Временную линию изменений через историю Git

Этот слой метаданных драматически увеличивает
то, что Cowork может делать значимо с InGit
Projects по сравнению с общими папками.

### 3.3. Git-Нативное Версионирование

Cowork может использовать Git через команды
оболочки. Но Cowork по сути не структурирует
работу как историю Git. Каждая сессия Cowork
может коммитить, может не коммитить. Сообщения
коммитов могут быть информативными, могут не
быть.

InGit обеспечивает **конвенции**, которые
делают историю Git значимой:
- Формат сообщения коммита
- Pre-commit хуки для валидации
- Стратегия ветвления для проектной работы
- Структура тегов для релизов
- Интеграция хуков со статусом задач

Это делает историю Git **навигируемой как
проектная история**, а не просто как лог
изменений кода.

### 3.4. Валидация и Контроль Качества

Cowork может создавать файлы с ошибками. У
общего Cowork нет доменных знаний для
валидации.

Хуки валидации InGit обеспечивают:
- YAML соответствует схеме
- Обязательные поля метаданных присутствуют
- Конвенции именования файлов соблюдены
- Большие файлы случайно не закоммичены
- Шифрование применено к чувствительным
  папкам

Если выставлены через MCP, Cowork использует
эти валидации автоматически. Выходы Cowork
становятся надёжно структурированными.

### 3.5. Шифрование для Чувствительного Содержания

Cowork имеет модель безопасности (изолированная
VM, разрешения), но не обеспечивает шифрование
содержания. Чувствительное содержание сидит
в открытом тексте (подчиняясь системе
разрешений).

Слой шифрования InGit (age, SQLCipher)
обеспечивает:
- Шифрование на уровне файла для
  чувствительных папок
- Проектное шифрование для целых проектов
- Управление ключами
- Дешифрование при доступе

Это важно для юридической работы, финансовых
данных, информации о здоровье — именно тех
категорий, с которыми Anthropic рекомендует
не использовать Cowork в настоящее время.

### 3.6. Offline-First Работа

Cowork требует, чтобы Claude Desktop работал.
Хотя Claude Desktop может работать с
локальными файлами, сам Cowork зависит от
инфраструктуры Anthropic.

InGit Projects работают полностью оффлайн.
Даже без Cowork структура Project, файлы,
история Git, валидация — всё работает. Когда
Cowork доступен, дополнительные способности;
когда нет, базовая линия по-прежнему
функциональна.

Это важно для: путешествий, локаций с плохой
связью, конфиденциально-чувствительной работы,
ситуаций, где зависимость от внешней
инфраструктуры нежелательна.

### 3.7. Переносимость Между AI-Платформами

Cowork является Anthropic-специфичным. Если вы
интенсивно используете Cowork и позже нужно
переключиться на платформы (Gemini, GPT,
локальные модели), Cowork-специфичные
паттерны не переносятся.

Структура InGit платформо-агностическая. Тот
же проект работает с любым AI, имеющим
соответствующий MCP сервер или плагин. Это
предотвращает привязку к конкретному
AI-провайдеру.

### 3.8. Миграция и Совместимость

Cowork имеет коннекторы к конкретным сервисам,
но не инструменты миграции для конвертации
между структурами проектов.

InGit может включать инструменты миграции:
- Импорт из Notion (сохранение структуры, где
  возможно)
- Импорт из Obsidian (сохранение связывания)
- Импорт из [GitHub](../docs/01-svyazi/03-component-catalog.md) Issues + Wiki
- Экспорт в стандартные форматы (Markdown +
  YAML)

Это делает InGit полезным как **хаб
совместимости** для пользователей, работающих
через множественные системы.

### 3.9. Что это означает

Ценность InGit при объединении с Cowork
сосредоточена в:
- Конвенциях структурированного рабочего
  пространства
- Метаданных как код
- Git-нативных конвенциях версионирования
- Инфраструктуре валидации
- Слое шифрования
- Offline работе
- Кросс-платформенной переносимости

Это именно качества «подложки». InGit — это
почва, на которой Cowork эффективно работает.
